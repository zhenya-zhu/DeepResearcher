# Deep Researcher 架构评估与优化方案

这份文档不是重复原始长文，而是直接回答三个工程问题：

1. 原方案哪些地方合理，哪些地方不适合当前环境。
2. 在 `restricts.md` 约束下，应该怎样重构成可运行方案。
3. 怎样把流程做成更健壮、可视化、容易 debug、又不被长上下文拖垮的系统。

## 先说结论

原始方案有方向感，但不够“工程化”。它更像一篇覆盖面很广的技术综述，而不是一份能直接指导当前仓库落地的设计。

保留的部分：

- 多阶段研究流程而不是单次检索问答，这个方向是对的。
- 把“计划、检索、综合、核查”拆开，而不是让一个 agent 一口气做完，思路是对的。
- 强调 checkpoint、补丁式编辑、重试和断路器，这些都属于生产系统必须项。
- 把引用核查单独拉出来，而不是默认相信写作模型，这一点也合理。

需要修改的部分：

- 模型绑定在 Claude 3.5/Claude 风格提示词上，已经过时，而且与本地 `restricts.md` 的多模型网关不匹配。
- 过度强调 LangGraph / 多智能体范式本身，却没有先定义状态对象、工件目录、失败恢复边界和调试接口。
- “Todo Files” 有启发，但如果没有统一 run 目录和结构化 checkpoint，很容易演化成难维护的文件堆。
- 原文把“显式思维链”“XML prompt 结构”写得过重，这对跨模型兼容性并不好，也不利于后续替换模型。
- 引用、证据、段落、来源之间的绑定关系没有收敛成清晰的数据模型，后面很难 debug“这句话从哪来”。
- 上下文控制没有真正落到实现策略，只是在理念层面提到了压缩和摘要。

## 必须接受的现实约束

来自 [restricts.md](/Users/I561043/SAP/DeepResearcher/restricts.md) 的关键约束不是附属信息，而应该成为架构的起点：

- LLM 统一入口应优先走 `http://localhost:6655/litellm/v1`，这样可以在一个 OpenAI-compatible 接口下切换 Anthropic、OpenAI、Gemini 模型。
- 默认全局限流应按 `16 RPM` 做，而不是让不同 agent 自己随便并发。
- 搜索优先使用 `ddgr`，而不是一开始就假设 Tavily / Firecrawl 之类外部服务一定可用。
- 外网请求应该默认支持 `http://proxy.sin.sap.corp:8080`。
- 不应该把 API key 硬编码进仓库。

这几条会直接决定代码结构：模型路由、搜索工具、限流器、网络抓取、日志和恢复机制都要围绕它们来写。

## 对原方案的核心判断

### 1. “多智能体”可以保留，但不能先把系统复杂化

原文默认往 Supervisor-Worker、多节点图编排上走，这在 paper 和 demo 里很常见，但对当前仓库不一定是第一优先级。

更合理的做法是：

- 先把流程建模成显式状态机。
- 角色可以逻辑上分成 planner / researcher / writer / verifier。
- 物理实现上不必先拆成复杂并发图。
- 先保证每个阶段输入输出可见、可重放、可恢复。

原因很简单：真正难 debug 的不是 agent 数量不够多，而是“某一轮研究后为什么状态变成这样”。如果没有稳定的状态快照和工件体系，多智能体只会让问题更难定位。

### 2. 不是不要 LangGraph，而是现在不该把它当第一步

LangGraph 的优势在于图编排、循环、分支和状态 reducer。问题在于：

- 当前目录里没有任何现成实现。
- 当前主要问题不是“图不够复杂”，而是“流程边界、证据归档、日志和限流都还没有工程化”。
- 对长时任务来说，先把 checkpoint/resume 和 artifact ledger 做好，价值比先引入外部编排框架更大。

所以建议是：

- 第一阶段：用显式 Python 状态机完成端到端流程。
- 第二阶段：如果后面确实需要并发 fan-out、人工插点、复杂条件边，就把现有状态对象平移到 LangGraph。

### 3. 模型路由必须从“模型中心”改成“角色中心”

原文有一个明显问题：它默认“底层大模型几乎就是 Claude”。

现在更合适的设计是按角色分配模型：

| 角色 | 目标 | 推荐主模型 | 推荐 fallback |
| :-- | :-- | :-- | :-- |
| Planner | 拆问题、生成研究大纲 | `anthropic--claude-4.6-sonnet` | `gpt-5`, `sonar-pro` |
| Researcher | 基于证据提炼 section finding | `anthropic--claude-4.6-sonnet` | `gpt-5`, `sonar-pro` |
| Writer | 报告成稿 | `anthropic--claude-4.6-sonnet` | `gpt-5`, `anthropic--claude-4.6-opus` |
| Verifier | 审稿、补洞、审计引用 | `anthropic--claude-4.6-sonnet` | `gpt-5`, `sonar-pro` |
| Fast / Utility | 轻量 query 改写、分类 | `anthropic--claude-4.5-haiku` | `gpt-5-mini`, `sonar` |

好处：

- 不把流程锁死在 Claude。
- 可以用不同模型家族互相校验，降低“同模型自证正确”的偏差。
- 模型失败时更容易降级，而不是整个系统一起卡死。

### 4. 上下文控制不能停留在“做摘要”，要变成硬规则

长时 Deep Research 最容易死在这里。原文提了压缩和记忆层次，但没有给出硬约束。

应该改成：

- 规划阶段只保留 `objective / sections / success_criteria / risks`。
- 研究阶段按 section 组织证据，不让一个 worker 看到全量网页文本。
- 每个 source 只保留有限 excerpt，而不是把全文直接塞回模型。
- 每轮结束只把 `section summary + findings + open questions` 带入下一轮。
- 最终写作阶段输入的是 section packet，而不是整个搜索历史。

这实际上是“证据账本优先”，而不是“聊天历史优先”。

## 优化后的目标架构

### 核心原则

- 本地网关优先：统一走 LiteLLM/OpenAI-compatible 接口。
- 角色化模型路由：按 planner / researcher / writer / verifier 分工。
- 显式状态机：先做清晰阶段，再谈复杂 agent 图。
- 证据账本：所有 claim 必须能追到 source_id。
- 可视化 trace：每个阶段都有日志、快照、prompt/response 工件。
- 可恢复：每轮、每 section、最终报告都落 checkpoint。
- 可降级：模型失败、抓取失败、搜索失败都允许 fallback，而不是整个 run 崩溃。

### 建议流程

```text
Question Intake
  -> Planning
  -> Section Research Loop
     -> Search (ddgr)
     -> Fetch / Normalize
     -> Evidence Pack
     -> Section Synthesis
  -> Gap Review
  -> Report Writing
  -> Audit / Citation Check
  -> Final Artifacts
```

### 每个阶段的最小输出契约

| 阶段 | 最小输出 |
| :-- | :-- |
| Planning | `objective`, `research_brief`, `sections[]`, `success_criteria`, `risks` |
| Search | `query`, `title`, `url`, `snippet` |
| Fetch | `raw_html`, `normalized_text`, `excerpt` |
| Section Synthesis | `summary`, `findings[]`, `open_questions`, `follow_up_queries`, `status` |
| Gap Review | `continue_research`, `global_gaps`, `focus_sections[]` |
| Report Writing | markdown report with `[source:S001]` style引用 |
| Audit | `status`, `issues[]` |

这样每个节点的输入输出都能落到磁盘上，而不是只有一串不可追踪的消息历史。

## 更适合 debug 的数据与工件模型

建议每次运行都生成独立 run 目录：

```text
runs/<run_id>/
  report.md
  events.jsonl
  trace.html
  summary.md
  checkpoints/
  state/
  sources/
  artifacts/prompts/
  artifacts/responses/
```

为什么这比 Todo Files 更适合当前项目：

- Todo 更适合任务管理，不一定适合细粒度证据追踪。
- run 目录天然和一次研究任务绑定，恢复和归档都更简单。
- `events.jsonl` 可以直接喂给后续前端，做 timeline 或 DAG 可视化。
- `trace.html` 让人不需要翻日志文件也能快速定位哪一步出错。

## 证据与引用设计

原始方案里 CitationAgent 的方向没有问题，但工程上建议简化为“两层”：

1. Section 级研究时，finding 必须携带 `source_ids`。
2. Final report 审计时，Verifier 检查是否存在无引用主张、引用过弱或覆盖不足。

这比一开始就追求复杂的 PDF 坐标映射更合适。原因是当前目录里还没有 UI，也没有稳定的文档解析链路。先把 claim-to-source 关系做扎实，比先做“点击高亮跳转”更重要。

## 健壮性设计

### 1. 全局限流而不是 agent 自治并发

`restricts.md` 已经明确给出 20 RPM 上限，建议实际限制到 16 RPM。

因此应该：

- 用全局 rate limiter 包住所有 LLM 调用。
- 需要并发时，只放开搜索和抓取，并保持 LLM 入口串行或小并发。
- 不要让多个子 agent 各自以为自己“还有配额”。

### 2. 模型断路器和 fallback

推荐每个模型维护独立 circuit breaker 状态：

- 连续失败 2 次后进入 cooldown。
- cooldown 时间内跳过该模型，直接切下一个候选。
- 在 trace 中明确记录跳过原因和失败计数。

### 3. 失败降级

建议的失败策略：

- 规划失败：退回到模板化 section。
- 抓取失败：保留搜索 snippet，继续。
- section synthesis 失败：写 heuristic summary，并把 source artifact 留给人看。
- audit 失败：不要吞掉错误，写入 `audit_issues`。

## 下一步建议

如果你后面要把它继续做成真正的生产级 Deep Research 系统，优先级建议如下：

1. 加入更稳定的内容抽取器，而不是只靠基础 HTML 提取。
2. 为 source/finding/report 建立更严格的 schema 校验。
3. 在 section 研究阶段加入更强的 query rewrite 和 source reranking。
4. 再考虑是否把显式状态机迁移到 LangGraph，以获得更强的图编排能力。
5. 有了稳定后端后，再建设前端化的 trace viewer、citation explorer 和人工插点界面。

## 最后判断

原设计“方向对，但形态不对”。

它的问题不是想得少，而是想得太散、太依赖某一家模型、太强调架构名词，反而削弱了当前环境里最重要的三件事：

- 真正受约束的模型与网络环境接入
- 可恢复、可检查的状态和证据工件
- 对长时任务的上下文削减与调试可见性

优化后的版本应该是：先把运行时系统做成透明的，再逐步把它演化成更复杂的 agent 网络。
