# DeepResearcher 自适应研究智能体重写计划

> CEO Review 于 2026-03-22 完成 | 分支: research/mar19 | 模式: SCOPE EXPANSION

## 背景与问题诊断

### 当前状态
在 `research/mar19` 分支上进行了 9 轮实验（exp1-exp9），评分从 45.4 提升到 86.0（+89%）。所有实验仅修改了 `prompts.py` 和 `config.py`，核心工作流 `workflow.py` 未做任何变更。

### 两个根本问题

**问题一：评估指标有缺陷**
- `evaluate.py` 中的 `compute_structural_score` 在报告字数超过参考报告 1.2 倍时惩罚分数
- 这是错误的——更充分的研究报告应该被奖励而非惩罚
- 评估只衡量表面结构（标题数、表格数、引用数），不衡量实际内容质量
- 仅用一个测试查询，代表性不足

**问题二：优化路径被人为限制**
- `program.md` 明确禁止修改 `workflow.py`
- 但 `workflow.py` 决定了核心研究逻辑：搜索轮次、gap review 策略、报告写作方式
- 只调 prompt 和 config 就像"在固定的流水线上打磨说明书"——天花板很低
- 86/100 分基本已是 prompt/config 调优的极限

## 愿景

### 10x 目标
将 DeepResearcher 从**线性管道**（plan→search→synthesize→repeat→write）转变为**自适应研究智能体**——能像资深分析师一样思考：动态调整研究深度、使用多种搜索策略、评估源质量、并在输出前自我审视修改。

### 理想体验
像拥有一位 24/7 工作的优秀初级分析师：
1. 先思考再行动——产出可审视的研究计划
2. 智能搜索——根据不同信息需求使用不同策略
3. 递进式理解——每轮搜索真正建立在前一轮基础之上
4. 深度分析写作——不是罗列事实，而是综合判断
5. 精确引用——每个声明有出处，出处有可信度评估
6. 美观输出——丰富的格式、适当的对比表、清晰的叙事弧

## 确定的范围（7 项扩展全部接受）

### 扩展 1: 自适应研究深度（动态轮次控制）
- **当前**：固定 3 轮搜索循环
- **目标**：基于证据充分度评分的动态控制——简单问题 1 轮即可，复杂问题可达 5-8 轮
- **核心变更**：gap review 从布尔决策（"继续吗？"）升级为量化评估（"该章节证据充分度 3/5，缺少 X 类信息"）
- **工作量**：CC ~1 小时

### 扩展 2: 多策略搜索
- **当前**：所有章节统一使用 DuckDuckGo 关键词搜索
- **目标**：根据信息需求类型选择搜索策略
  - 官方文档：`site:openai.com deep research`
  - 开源代码：`site:github.com deep-research`
  - 学术论文：`site:arxiv.org deep research agent`
  - 链接追踪：从高质量页面中提取并跟踪相关链接
- **核心变更**：搜索模块新增策略选择器，planner 为每个 section 指定搜索策略
- **工作量**：CC ~1 小时

### 扩展 3: 源可信度评分
- **当前**：所有源一视同仁——随机博客与官方文档权重相同
- **目标**：为每个源计算可信度评分
  - 域名信誉（openai.com > random-blog.xyz）
  - 内容类型（官方文档 > 分析博客 > 论坛讨论）
  - 时效性（2025年 > 2022年）
- **影响**：报告生成时优先引用高可信度源，对低可信度源标注不确定性
- **工作量**：CC ~1 小时

### 扩展 4: 自我审视循环（写作→批评→修改）
- **当前**：每个 section 写一次就定稿；audit 只记录问题不修改
- **目标**：
  1. Writer 写出初版 section
  2. Critic 审查：论点有证据支撑吗？逻辑链完整吗？遗漏了哪些重要视角？
  3. Writer 根据 critique 修改并输出终版
- **原理**：类似 Google DeepMind Aletheia 系统的 Generator-Verifier-Reviser 三步走
- **成本**：token 消耗 +30%，换取显著的分析质量提升
- **工作量**：CC ~45 分钟

### 扩展 5: 评估体系重构
- **修复长度惩罚**：移除 1.2x 超长惩罚，改为"充分度奖励"
- **增加语义评分**：报告是否覆盖了参考报告的主要主题
- **多查询测试**：使用 queries.json 中的全部 3 个查询而非 1 个
- **LLM 评判**：评估"报告是否真正回答了问题"
- **工作量**：CC ~45 分钟

### 扩展 6: 跨章节综合与智能资源分配
- **当前**：各章节独立研究，互不引用，像"独立摘要的拼接"
- **目标**：
  - 写报告前增加"跨章节综合"步骤
  - 识别各章节间的重复信息和共同主题
  - 智能资源分配：证据充分的 section 少搜，证据薄弱的多搜
  - 消除矛盾：如果两个章节对同一事实描述不一致，主动解决
- **工作量**：CC ~2 小时

### 扩展 7: 研究透明度（可读追踪）
- **当前**：RunArtifacts 系统以 JSON 日志存储，不可读
- **目标**：运行结束时自动生成 `research-trace.md`
  - 研究计划及每轮决策记录
  - 每个源的可信度评分
  - 失败搜索的记录和原因
  - 证据充分度判断过程
- **工作量**：CC ~30 分钟

## 实现方式

**增量重写** `workflow.py`：
- 保留现有工具函数（source 注册、URL 去重、section draft 生成等）
- 重写核心控制流（研究循环、gap review、报告生成）
- 保持 API 兼容（`DeepResearcher.run()` 接口不变）
- 保留所有 tracing/artifact 基础设施

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| Token 成本大幅增加 | 高 | 中 | 设置每次运行的 token 预算上限；自适应深度避免简单问题过度搜索 |
| LLM 调用失败（充分度评估等） | 中 | 低 | 所有新增 LLM 调用都有合理的降级策略 |
| 重写引入回归 | 低 | 高 | 增量重写 + 每步 git commit + 3 查询测试验证 |
| 域名可信度评分的误判 | 中 | 低 | 使用已知高质量域名白名单而非纯启发式 |

## 工程审查后新增范围

### 扩展 8: 并行章节研究（异步）
- **当前**：各章节按顺序逐一研究
- **目标**：同一轮次内独立章节并行研究，共享速率限制器
- **约束**：HAI Proxy 限制 20 req/min/user，并行受限于速率而非并发
- **策略**：搜索/抓取（网络 I/O）并行，LLM 调用通过现有 IntervalRateLimiter 串行
- **工作量**：CC ~1 小时

### 扩展 9: 链接追踪（引用链跟踪）
- **当前**：仅使用搜索引擎返回的直接结果
- **目标**：当高可信度源引用其他 URL（博客→论文，论文→论文），跟踪这些链接以丰富证据基础
- **约束**：设置深度限制（最多 1 层跟踪），环检测避免无限循环
- **依赖**：扩展 2（多策略搜索）+ 扩展 3（源可信度评分）
- **工作量**：CC ~30 分钟

## 工程审查关键决策

1. **方法级扩展**而非控制流重写——现有架构已支持所有扩展作为方法级注入
2. **prompts.py 保持单文件**——~650 行可管理，避免导入链变更
3. **源可信度使用启发式评分**——域名白名单 + URL 模式 + 日期提取，不使用 LLM
4. **完整测试覆盖**——新增 ~15 个测试用例（3 个新测试文件 + 2 个扩展文件）
5. **evaluate.py 零测试是关键缺陷**——修改评分逻辑前必须先补测试

## 未纳入范围

- 真正的浏览器 agent（Playwright/Puppeteer）——属于 "ocean"，不在本次 "lake" 范围
- 多模型策略（便宜模型搜索、贵模型综合）——可作为后续优化
- 完整 i18n 支持——当前中英文混合处理已够用
- Prompt 拆分为多文件——单文件 650 行可管理
- Prompt caching（已记录到 TODOS.md）——可独立实施
